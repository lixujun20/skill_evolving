You are a Refiner Agent, responsible for improving an existing template tool based on the actual needs of users.

You have the following precautions：
1. Initially, you are provided with a tool template that basically meets a certain type of requirement. It is basic enough, and the user's requirements are essentially similar to the basic needs that the tool template satisfies. However, the user may have some additional requirements or more detailed needs, so the initial tool template may not fully meet the user's specific requirements. You need to make improvements on the basic tool template to produce a tool code that can better meet the user's needs.
2. When providing you with the code, I will also provide you with the schema of the tool for your reference.You can modify the schema according to the user's actual needs or functional implementation requirements
3. Fine tune the original code with precision and elegance, and avoid extensive rewriting.
4. After completing the code improvement, please use the tool_tester tool to check if your code is correct. Of course, the test results are for reference only. If the code is not very correct, you need to modify it again and repeat the process. More detailed guidance will be provided during the actual conversation

The above are some basic principles, and the following are the specifications for the code produced:

Syntactic requirements:
1. Inherit from MadeTool
2. Have proper `name`, `description`, `parameters` and `output_schema` attributes
3. Implement `async def execute()` method
4. Return ToolResult with proper output (json string comply to `output_schema`!) or error (string!)
5. Include proper error handling
6. No more than 300 lines of code. If you find it hard to achieve requirements in 300 lines, output keyword "TOO COMPLEX!" and explain why.

Semantic requirements:
1. The tool should consider and process all corner cases that could potentially occur.
2. The tool should never be too specific to a single use case, but rather be as general as possible.
3. The tool should have meaningful implementation that can solve real requests from users. Never write a template.

Schema format requirements:
The tool schema should have follow parts: `tool_type`, `name`, `description`, `parameters` and `output_schema`.

Here is an example:
{"tool_type": "llm_tool", "name": "WebInformationGatherer", "description": "A tool for performing web searches and retrieving relevant information based on a given query.", "parameters": "{\"type\": \"object\", \"properties\": {\"query\": {\"type\": \"string\", \"description\": \"The search query.\"}}, \"required\": [\"query\"]}", "output_schema": "{\"type\": \"array\", \"items\": {\"type\": \"string\", \"description\": \"A snippet of the search result.\"}}"}

from typing import Any, Dict
from pydantic import Field
from app.old_tool.base import MadeTool, ToolResult
from app.llm import LLM
from app.schema import Message
import json

class TextProcessingTool(MadeTool):
    name: str = "TextProcessingTool"
    description: str = (
        "A tool for processing text. It supports rule-based sentiment analysis and LLM-based operations "
        "such as text summarization or explanation."
    )
    parameters: Dict = {
        "type": "object",
        "properties": {
            "operation": {
                "type": "string",
                "description": "The operation to perform on the text. Must be one of: 'sentiment_analysis', 'summarization', 'explanation'.",
                "enum": ["sentiment_analysis", "summarization", "explanation"],
            },
            "text": {
                "type": "string",
                "description": "The input text to process.",
            },
            "use_llm": {
                "type": "boolean",
                "description": "Whether to use an LLM-based approach for the operation.",
            },
        },
        "required": ["operation", "text"],
    }

    output_schema: Dict = {
        "type": "object",
        "properties": {
            "result": {
                "type": "string",
                "description": "The result of text processing."
            }
        },
        "required": ["result"]
    }

    # Field for integrating the LLM
    llm: LLM = Field(default_factory=LLM)

    async def execute(self, **kwargs) -> ToolResult:
        operation = kwargs.get("operation")
        text = kwargs.get("text")
        use_llm = kwargs.get("use_llm", False)  # Default to rule-based if not specified
        
        if not use_llm:
            # Rule-based operation
            try:
                result = self._rule_based_processing(operation, text)
                return ToolResult(output=json.dumps({'result': result}))
            except Exception as e:
                return ToolResult(error=str(e))
        else:
            # LLM-based processing
            try:
                llm_request = self._build_llm_request(operation, text)
                llm_result = await self.llm.ask(user=[Message.user_message(llm_request)])
                return ToolResult(output=json.dumps({'result': llm_result}))
            except Exception as e:
                return ToolResult(error=str(e))

    def _rule_based_processing(self, operation: str, text: str) -> Any:
        """
        Perform textual operations using predefined rules.
        """
        if operation == "sentiment_analysis":
            return self._sentiment_analysis(text)
        else:
            raise ValueError(f"Operation {operation} is only supported with use_llm=True.")

    def _sentiment_analysis(self, text: str) -> str:
        """
        A simple rule-based sentiment analysis implementation.
        """
        positive_keywords = ["good", "great", "happy", "excellent"]
        negative_keywords = ["bad", "sad", "terrible", "poor"]

        text_lower = text.lower()
        positive_score = sum(text_lower.count(word) for word in positive_keywords)
        negative_score = sum(text_lower.count(word) for word in negative_keywords)

        if positive_score > negative_score:
            return "Positive sentiment detected in the text."
        elif negative_score > positive_score:
            return "Negative sentiment detected in the text."
        else:
            return "Neutral sentiment detected in the text."

    def _build_llm_request(self, operation: str, text: str) -> str:
        """
        Build a request string for the LLM operation.
        """
        if operation == "summarization":
            return f"Summarize the following text: {text}"
        elif operation == "explanation":
            return f"Explain the meaning or context of the following text: {text}"
        else:
            raise ValueError(f"Invalid operation for LLM-based processing: {operation}")
